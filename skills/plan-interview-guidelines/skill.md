# Planner Interview Guidelines

Core domain knowledge for generating structured interview questions that systematically assess **planner (기획자)** candidate competency across multiple dimensions. All agents in the plan-interview pipeline MUST apply these rules.

---

## 1. Ice Breaking Rules

**Purpose:** Rapport formation + communication style assessment. NOT competency evaluation.

**Count:** 2-4 questions per interview.

**Selection Criteria:**

| Criterion | Rule |
|-----------|------|
| Low Cognitive Load | No right answers, no deep thinking required |
| Interest Based | Use resume hobbies, certifications, recent interests |
| Contextual | Arrival, weather, simple greetings — universally easy |
| Domain Fit (optional) | If candidate has relevant industry experience, lightly reference it |

**Example Patterns (adapt to each candidate):**
- "오늘 오시는 길은 찾기 어렵지 않으셨나요?"
- "이력서에 [취미/특기]가 눈에 띄던데, 최근에도 즐기실 시간이 있으셨나요?"
- "긴장을 푸는 본인만의 방법이 있으신가요?"
- "최근에 업무 외적으로 가장 몰입했던 즐거움은 무엇이었나요?"
- "대외활동/채용 시장에 관심을 갖게 된 계기가 있으신가요?" *(domain-fit: use when candidate has relevant industry background)*

## 2. Six Perspectives Framework (Planner-Adapted)

The core analytical framework for deep-dive questions. Each project section MUST cover **minimum 4 of 6** perspectives.

### 2.1 Why — 문제/기회 식별 및 정의

**Assesses:** Problem sensing, opportunity identification, user pain point awareness, market context understanding

**Probe patterns:**
- "이 기능/프로젝트를 기획하게 된 배경은 무엇인가요? 어떤 문제를 해결하려 했나요?"
- "그 문제가 존재한다는 것을 어떻게 파악하셨나요? (사용자 피드백, 데이터 분석, 시장 조사 등)"
- "해결하려는 문제의 대상 사용자는 누구이며, 그들의 핵심 니즈를 어떻게 정의하셨나요?"
- "왜 그 시점에 이 문제를 해결하는 것이 중요했나요? 다른 이슈들보다 우선순위가 높았던 이유는?"

**Signal Integration — Domain Understanding:** When candidate has 대외활동/채용 플랫폼 experience, probe domain-specific problem identification: "링커리어 사용자들의 어떤 특성이 이 문제 정의에 영향을 미쳤나요?"

### 2.2 Why Not / Trade-off — 스코프 및 우선순위 결정

**Assesses:** Alternative evaluation, feature prioritization rationale, scope decisions, project-level prioritization

**Probe patterns:**
- "이 기능의 스코프를 정할 때, 의도적으로 제외한 것은 무엇인가요? 왜 제외했나요?"
- "다른 접근 방식이나 솔루션도 고려하셨나요? 최종 방향을 선택한 결정적 근거는?"
- "MVP 범위를 어떻게 정의하셨나요? 무엇을 넣고 무엇을 뺄지 어떤 기준으로 결정했나요?"
- "이 기능을 추가함으로써 포기하거나 미루게 된 다른 기획 건이 있었나요?"

**Signal Integration — Project Prioritization Insight:** Expand to project-level decisions:
- "여러 기획 건 중 왜 이 프로젝트가 우선순위가 높았나요? 어떤 프레임워크나 기준을 사용했나요?"
- "우선순위가 변경된 적이 있나요? 어떤 상황에서, 어떻게 대응하셨나요?"
- "디퍼(defer)된 기획 건은 어떻게 관리하셨나요?"

### 2.3 Stakeholder & Communication — 이해관계자 정렬 및 커뮤니케이션

**Assesses:** Cross-functional alignment, persuasion ability, conflict resolution, communication clarity

**Probe patterns:**
- "이 기획을 진행하면서 관련된 이해관계자(개발, 디자인, 사업, 경영진)는 누구였나요?"
- "이해관계자 간 의견이 충돌했던 경험이 있나요? 어떻게 합의를 이끌어내셨나요?"
- "개발팀에 기획 의도를 전달할 때, 어떤 방식으로 커뮤니케이션하셨나요? (PRD, 와이어프레임, 구두 등)"
- "기획안이 반려되거나 큰 수정을 요구받은 경험이 있나요? 어떻게 대응하셨나요?"

### 2.4 How / Who — 실행 과정 및 기여도

**Assesses:** Planning process depth, artifact quality, personal contribution vs team boundary, cross-functional coordination

**Probe patterns:**
- "기획 과정에서 구체적으로 어떤 산출물을 만드셨나요? (PRD, 유저 스토리, 와이어프레임, 플로우차트 등)"
- "기획부터 출시까지의 전체 프로세스에서 본인이 주도한 부분과 다른 팀원/조직이 담당한 부분의 경계는?"
- "기획 과정에서 가장 어려웠던 의사결정은 무엇이었고, 어떻게 해결하셨나요?"
- "디자이너/개발자와 협업할 때 기획 의도가 제대로 반영되었는지 어떻게 확인하셨나요?"

### 2.5 Validation / Risk — 사용자 검증 및 리스크 관리

**Assesses:** Assumption validation method, user research depth, risk identification and mitigation

**Probe patterns:**
- "기획 단계에서 사용자 니즈에 대한 가설을 어떻게 검증하셨나요? (사용자 인터뷰, 설문, A/B 테스트, 프로토타입 등)"
- "출시 전에 예상했던 리스크는 무엇이었고, 어떻게 대비하셨나요?"
- "예상과 다른 사용자 반응이 나온 적이 있나요? 어떻게 대응하셨나요?"
- "데이터가 부족한 상황에서 의사결정을 해야 했던 경험이 있나요? 어떤 근거로 판단하셨나요?"

### 2.6 So What — 성과 측정, KPI 및 회고

**Assesses:** Impact measurement ability, KPI definition and tracking, retrospective insight, iteration capability

**Probe patterns:**
- "이 기획의 성공/실패를 어떤 KPI 지표로 판단하셨나요? 그 KPI를 선택한 이유는?"
- "기획 의도대로 성과가 나왔는지 어떻게 측정하셨나요? 구체적인 수치가 있나요?"
- "KPI 목표를 달성하지 못했을 때 어떻게 대응하셨나요? 어떤 개선 작업을 했나요?"
- "지금 다시 그 시점으로 돌아간다면, 아쉬웠던 점이나 다르게 기획하고 싶은 부분이 있나요?"
- "출시 이후 사용자 피드백을 바탕으로 어떤 후속 기획을 진행하셨나요?"

**Signal Integration — KPI Awareness:** KPI definition is distinct from generic outcome reporting. Probe the PROCESS of defining KPI, not just the numbers:
- "KPI를 직접 정의하셨나요, 아니면 조직에서 주어진 것인가요?"
- "KPI 측정을 위한 데이터 수집/트래킹 체계는 어떻게 구축하셨나요?"

### Coverage Rule

| Constraint | Value |
|-----------|-------|
| Minimum perspectives per project | 4 of 6 |
| Must-include for user-facing features | Why + Validation + So What |
| Must-include for scope/priority decisions | Why Not/Trade-off |
| Must-include for cross-functional projects | Stakeholder & Communication + How/Who |
| Must-include for KPI-driven initiatives | So What (with KPI probes) |

## 3. Career Level Strategy

**Rule:** Determine career level FIRST from resume/portfolio, then apply corresponding strategy to ALL questions.

### 3.1 Junior (Entry Level Planner)

**Keywords:** Problem-sensing, User Empathy, Structured Thinking, Execution Speed

**Analysis Points:**
- **Problem sensitivity:** Ability to identify user problems beyond surface-level symptoms
- **Structured approach:** Organizes thinking logically (vs jumping to solutions without analysis)
- **User empathy:** Genuine interest in understanding users, not just following instructions
- **Execution follow-through:** Takes ownership of a feature from idea to launch, handles details

**Question Strategy (Growth Focus):**
- **Problem-solving process:** "기획 과정에서 막막했던 순간이 있었나요? 어떻게 실마리를 찾아가셨나요?"
- **User understanding:** "사용자 입장에서 가장 고민이 됐던 포인트는 무엇이었고, 어떻게 해결하셨나요?"
- **Collaboration attitude:** "개발팀이나 디자인팀과 의견이 달랐을 때 어떻게 조율하셨나요?"
- **Learning from failure:** "기획한 기능이 기대만큼 성과가 나오지 않았던 경험이 있나요? 어떻게 대처하셨나요?"

**KPI Integration (Hygiene level):** Junior must understand the CONCEPT of KPI and success metrics.
- "이 기능의 성공을 어떤 기준으로 판단하셨나요?" (can articulate = Pass, cannot = Fail)

**Prioritization Integration (Basic level):** Junior must show awareness of prioritization.
- "여러 기능 요청 중 이것을 먼저 진행한 이유는 무엇인가요?"

### 3.2 Senior (Experienced Planner)

**Keywords:** Strategic Thinking, Cross-functional Leadership, Data-driven Decisions, Stakeholder Management

**Analysis Points:**
- **Strategic prioritization:** Choosing the right thing to build when everything seems important
- **Cross-functional influence:** Aligning diverse stakeholders without formal authority
- **Data-driven iteration:** Using metrics to validate, pivot, or kill features
- **Organizational impact:** Improving team processes, establishing planning frameworks

**Question Strategy (Impact Focus):**
- **Strategic trade-offs:** "여러 기획 건의 우선순위를 어떻게 조율하셨나요? 사업적 긴급성과 사용자 가치가 충돌할 때는?"
- **Stakeholder persuasion:** "경영진이나 사업부의 요구와 사용자 니즈가 상충할 때 어떻게 설득하셨나요?"
- **Data-driven pivot:** "출시 후 데이터를 보고 기획 방향을 크게 수정한 경험이 있나요? 그 과정은?"
- **Process improvement:** "기획 프로세스 자체를 개선하거나 새로운 프레임워크를 도입한 경험이 있나요?"

**KPI Integration (Core/Advanced level):** Senior must have DESIGNED KPI systems and used them for decision-making.
- "KPI 체계를 직접 설계하신 경험이 있나요? 어떤 지표를 어떻게 선정하셨나요?"
- "KPI 기반으로 기획 방향을 변경한 구체적인 사례가 있나요?"

**Prioritization Integration (Advanced level):** Senior must demonstrate strategic multi-project prioritization.
- "동시에 진행 중인 여러 기획 건의 우선순위를 어떤 프레임워크로 정하셨나요?"
- "리소스 제약 상황에서 기획 건을 줄이거나 디퍼(defer)한 경험과 그 기준은?"

## 4. Portfolio Signal Analysis

Read between the lines of planner resumes. Detect these 8 signals and design questions to probe them.

### Signal 1: "Facilitated" vs "Drove"

| Pattern | Example | Question Strategy |
|---------|---------|-------------------|
| Facilitated (coordination) | "프로젝트를 관리했다", "일정을 조율했다" | Ask: What decisions did YOU make? What was your planning authority? |
| Drove (ownership) | "사용자 조사 기반으로 기능 A를 제안하고 출시까지 리드" | Ask: What was the hardest stakeholder pushback? How did you handle it? |

**Rule:** Focus questions on DECISION-MAKING ownership, not project management coordination.

### Signal 2: Metric Attribution

| Check | Red Flag | Probe |
|-------|----------|-------|
| Business impact | "MAU 200% 성장" without causal link | "이 성장에 본인의 기획이 구체적으로 어떻게 기여했나요?" |
| Revenue claims | "매출 N억 달성" in junior role | "매출에 기여한 여러 요인 중 본인의 기획이 차지하는 비중은?" |
| User metrics | Specific numbers without measurement context | "이 수치는 어떤 도구/방법으로 측정하셨나요?" |

**Rule:** If business metrics seem org-level rather than personally attributable, design questions to clarify the causal chain.

### Signal 3: Decision Ownership

| Claim | Probe Strategy |
|-------|---------------|
| "기획을 담당했다" (vague) | Ask for specific decisions: "기획 과정에서 본인이 직접 결정한 것과 상위 결재가 필요했던 것의 경계는?" |
| "요구사항 정의 100%" | Ask about scope: "요구사항 중 본인이 새로 발굴한 것 vs 이미 정해진 것의 비율은?" |
| "로드맵 수립" | Ask about authority: "로드맵의 우선순위를 최종 결정한 사람은 누구인가요?" |

**Rule:** Probe decision ownership by asking about specific DECISION BOUNDARIES, not general responsibilities.

### Signal 4: User Research Evidence

| Pattern | Assessment | Probe |
|---------|-----------|-------|
| "사용자 조사 진행" (generic) | Shallow — method unknown | "어떤 방법론으로, 몇 명의 사용자를 대상으로 조사했나요?" |
| "인터뷰 N회, 설문 N명" (specific) | Deep — quantified effort | "인터뷰에서 가장 예상 밖이었던 발견은 무엇인가요?" |
| No user research mentioned | Gap — assumed user needs | "사용자 니즈를 어떻게 파악하셨나요? 직접 확인한 건가요?" |

**Rule:** Distinguish between assumed user needs and validated user needs. Design questions to surface the validation process.

### Signal 5: Data Literacy Signal

| Evidence | Assessment | Probe |
|----------|-----------|-------|
| "A/B 테스트 진행" | Active data use | "A/B 테스트 설계 시 가설과 성공 기준은 어떻게 정하셨나요?" |
| "퍼널 분석으로 이탈 구간 발견" | Analytical depth | "발견한 이탈 구간에 대해 어떤 가설을 세우고 어떻게 검증했나요?" |
| Metrics mentioned without analysis context | Passive data exposure | "이 수치를 보고 어떤 액션을 취하셨나요?" |
| No data-related activity | Data gap | "의사결정 시 데이터를 활용한 경험이 있나요?" |

**Rule:** Assess whether the candidate USES data to make decisions, not just reports data.

### Signal 6: KPI Awareness

| Evidence | Assessment | Probe |
|----------|-----------|-------|
| KPI explicitly defined in portfolio | Strong — proactive measurement | "이 KPI를 선택한 근거는 무엇인가요? 다른 후보 지표는 없었나요?" |
| Results mentioned without KPI context | Moderate — results-aware but process unclear | "이 기획의 성공/실패를 어떤 KPI 지표로 판단하셨나요?" |
| No success criteria mentioned | Weak — outcome measurement gap | "이 기능이 성공적이었는지 어떻게 알 수 있나요?" |

**Rule:** KPI definition capability is a core planner competency. If missing, design at least 1 question to probe it.

### Signal 7: Project Prioritization Insight

| Evidence | Assessment | Probe |
|----------|-----------|-------|
| Prioritization framework mentioned | Strong — systematic approach | "그 프레임워크를 선택한 이유는? 적용 시 어려웠던 점은?" |
| Multiple projects with timeline context | Moderate — awareness shown | "여러 기획 건 중 왜 이 프로젝트가 우선순위가 높았나요?" |
| Single project focus only | Unclear — cannot assess | "동시에 다른 기획 건도 있었나요? 우선순위는 어떻게 정했나요?" |

**Rule:** For Senior candidates, prioritization rationale is a critical assessment dimension. For Junior, basic awareness is sufficient.

### Signal 8: Domain Understanding (링커리어)

| Evidence | Assessment | Probe |
|----------|-----------|-------|
| 대외활동/채용/교육 플랫폼 경험 | High relevance — domain overlap | "이전 경험에서 링커리어와 유사한 사용자 세그먼트를 다뤄본 적이 있나요?" |
| B2C 플랫폼/커뮤니티 경험 | Moderate relevance — transferable | "플랫폼 사용자의 양면(공급/수요) 특성을 기획에 어떻게 반영하셨나요?" |
| Domain overlap 없음 | Low relevance — assess adaptability | "새로운 도메인에 적응할 때 어떤 방식으로 사용자를 이해하시나요?" |

**Rule:** Domain understanding assesses PLANNING COMPETENCY within domain context, NOT domain knowledge quiz. Focus on how the candidate applies planning skills to understand new/similar domains.

## 5. Question Format Standard

### 5.1 Document Structure

All interview documents MUST follow this structure:

```
1. Candidate Profile Summary (5 lines max)
2. Ice Breaking (2-4 questions)
3. Project A Section (3-6 questions)
4. Project B Section (3-6 questions)
5. [Optional] Common Competency Section
6. Interview Timing Guide Table
7. Quality Checklist
```

### 5.2 Grouping Rule

**CRITICAL:** All questions MUST be grouped by project.
- Validation/risk questions belong INSIDE the project section, NOT in a separate section.
- No "additional questions" or "verification" sections at the end.
- Interviewer should be able to go project-by-project sequentially.

### 5.3 Per-Question Format

```markdown
### Question Title (symbolic/evocative Korean title)
- **연관 기능 (Related Feature):** Specific feature/initiative this question connects to
- **빌드업 질문 (Buildup Question):** Easy entry point — "~~ 기능을 기획하실 때 가장 고민했던 지점은 무엇인가요?"
- **핵심 질문 (Deep Dive Question):** 1-3 probing questions on planning depth, decisions, trade-offs
- **평가 기준 (Evaluation Criteria):**
  - **[Hygiene Check]:** Pass (OK) — minimum requirement / Fail (Red Flag) — disqualifying signal
  - **[Core Competency]:** Plus (+) — practical depth indicator / Minus (-) — shallow understanding
  - **[Advanced Insight]:** Plus (+) — senior-level insight / Minus (-) — limited perspective
```

### 5.4 Buildup → Deep Dive Flow

Follow-up questions should flow: **problem definition → alternatives → stakeholder alignment → impact measurement**

| Step | Pattern |
|------|---------|
| 1. Problem definition | "이 기획의 배경과 해결하려는 문제는 무엇이었나요?" |
| 2. Alternatives | "다른 접근 방식도 고려하셨나요? 왜 이 방향을 선택했나요?" |
| 3. Stakeholder alignment | "관련 이해관계자들과 어떻게 합의를 이루셨나요?" |
| 4. Impact measurement | "기획의 성과를 어떻게 측정했고, 결과는 어땠나요?" |

### 5.5 Evaluation Criteria Levels

| Level | Purpose | Format | Applies To |
|-------|---------|--------|-----------|
| Hygiene Check | Basic planner competency | Binary: Pass / Fail | Problem articulation, user awareness, basic metrics, KPI concept |
| Core Competency | Practical planning depth | Gradient: Plus / Minus | Data-driven reasoning, stakeholder management, structured prioritization, user validation, KPI-based decisions |
| Advanced Insight | Senior-level strategic thinking | Gradient: Plus / Minus | Market positioning, cross-functional influence, product vision, KPI system design, portfolio-level prioritization |

### 5.6 Timing Guide

| Section | Duration | Notes |
|---------|----------|-------|
| Ice Breaking | 3-5 min | Rapport building |
| Main Project A | 15-20 min | Deepest product deep dive |
| Main Project B | 10-15 min | Second priority |
| Common/Wrap-up | 5-10 min | Collaboration, learning, closing |
| **Total** | **~30 min** | |

### 5.7 Quality Checklist (must pass before delivery)
